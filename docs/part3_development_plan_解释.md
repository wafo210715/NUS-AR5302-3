## 一、Part 3 到底要做什么？（一句话版）

**用地图上的"兴趣点"数据（比如餐厅、商店、学校），通过一个叫 LDA 的算法，自动给新加坡每个公交站/地铁站打上"功能标签"。**

举个例子：
- Orchard 站附近有很多餐厅和商店 → 算法会把它标记为 **"餐饮零售区"**
- NUS Kent Ridge 站附近有很多学校和图书馆 → 算法会把它标记为 **"教育区"**
- CBD 站附近有很多银行和写字楼 → 算法会把它标记为 **"商业办公区"**

为什么要做这件事？因为 Part 1 已经知道"NUS 的人去了 Orchard"，但光知道目的地名字不够，我们需要知道 Orchard 是**什么类型的地方**，才能回答研究问题："大学社区的出行目的地有什么特征？"

## 二、LDA 是什么？（用人话解释）

LDA 全名叫 **Latent Dirichlet Allocation**（隐狄利克雷分布），听起来很吓人，但其实很好理解。

### 类比：猜一本书的主题

假设你从来没读过一本书，只能看到里面出现的词：

- 书 A 里有：`猫` `狗` `鱼` `仓鼠` `兽医` `粮` ...
- 书 B 里有：`法院` `律师` `判决` `合同` `证据` ...

你肯定会猜：书 A 是关于**宠物**的，书 B 是关于**法律**的。

你并不知道书名，但你通过**哪些词出现了、出现了多少次**，推断出了书的主题。

**LDA 就是让电脑做同样的事。**

### 映射到我们的项目

| 书的类比 | 我们的项目 | 具体例子 |
|---------|-----------|---------|
| 一本书 | 一个交通站点 | Orchard MRT 站 |
| 书里的词 | 站点附近的兴趣点类别 | 餐厅、商店、银行、学校... |
| 书的主题 | 站点的城市功能 | 餐饮零售区、商业办公区... |
| 一类词在书里出现的频率 | 一类 POI 在站点附近出现的数量 | Orchard 附近有 50 家餐厅、3 家银行... |

LDA 会自动分析所有站点的 POI 组成，找出几个"主题"，每个主题就是一组经常一起出现的 POI 类型。

### 为什么选 LDA 而不是别的方法？

组里研究过 4 篇论文，比较了多种方法：

| 方法 | 为什么不用 |
|------|-----------|
| Doc2Vec | 需要"标准答案"来验证，我们没有 |
| CLIP（图像识别） | 需要街景照片，新加坡覆盖率不够 |
| 简单计数 | 太蠢了，无法区分"50家餐厅+1家医院"和"30家餐厅+21家医院"的区别 |
| **LDA** | **不需要标准答案，能自动发现规律，结果好解释** ✓ |

## 三、数据从哪来？

### OpenStreetMap（OSM）

OSM 就像一个**免费的、全世界志愿者一起编辑的地图**——类似"维基百科版的 Google Maps"。

任何人都可以在 OSM 上添加一个餐厅、一家商店、一个公交站的位置和类别信息。比如：

```json
{
  "name": "肯德基（乌节路店）",
  "amenity": "fast_food",
  "cuisine": "fried_chicken",
  "latitude": 1.3048,
  "longitude": 103.8318
}
```

### OSM 的优缺点

**优点：**
- 免费，不需要申请 API key
- 全球覆盖，数据量大
- 标签丰富（一个餐厅可以标注菜系、营业时间、是否有停车位等）

**缺点（很重要！）：**
- **完整度只有约 28%**（Yeow 2021 的研究）——意思是新加坡真实存在的 POI 中，只有约 1/4 被录入了 OSM
- 商业区（如 Orchard）录得比较全，住宅区和工业区录得很少
- 24% 的 POI 连名字都没有
- 没有精确的"创建时间"（只有"最后编辑时间"）

**应对策略：** 我们在论文里会坦诚地说"我们分析的是 OSM 上**已录入的**城市环境"，而不是完整城市环境。这是学术上可以接受的表述。

### 两个 API 的分工

我们会用到两个不同的 API 来获取 OSM 的数据：

| | Overpass API | ohsome API |
|--|-------------|-----------|
| **是什么** | OSM 的"查询接口" | OSM 的"统计接口" |
| **能做什么** | 下载具体每个 POI 的详细信息（名字、坐标、类别等） | 查看聚合统计（"新加坡总共有多少个餐厅？随时间怎么变化的？"） |
| **打个比方** | 去超市把每件商品都看一遍 | 看超市的月度销售报表 |
| **用在** | Task 1（下载 POI） | Task 2（数据质量分析） |

### Task 1：下载全新加坡的 POI 数据

**文件：** `utils/download_pois.py`（Python 脚本，新建）

**在干什么：**
从 OSM 下载新加坡所有的兴趣点（餐厅、商店、学校、医院等），保存为一个 CSV 文件。

**具体怎么做的：**
1. 定义新加坡的地理范围（一个矩形框：北纬 1.16° 到 1.47°，东经 103.60° 到 104.08°）
2. 把这个矩形分成 4 个小块来查询（避免一次查太多导致超时）
3. 对每一块，向 Overpass API 发送查询请求，要 8 种类别的 POI：
   - `amenity`（公共设施：餐厅、学校、医院、银行...）
   - `shop`（商店：便利店、服装店、超市...）
   - `leisure`（休闲：公园、健身房、游乐场...）
   - `tourism`（旅游：酒店、景点、博物馆...）
   - `office`（办公：公司、律师事务所...）
   - `healthcare`（医疗：诊所、牙医、药房...）
   - `craft`（手工艺：裁缝、修鞋、钥匙店...）
   - `historic`（历史：纪念碑、古迹...）
4. 每次查询之间等 5 秒（礼貌，不要轰炸别人的服务器）
5. 如果查询失败了，自动重试最多 3 次

**输出：** `data/sg_pois_all.csv`——一个 CSV 表格，每行是一个 POI，包含它的坐标、类别、名字、版本号、编辑者等信息。

**预期结果：** 大概 10 万到 20 万个 POI。

**怎么验证下载成功：**
- 总数在合理范围内
- 8 个类别都有数据（不能某个类别是 0）
- 坐标都在新加坡范围内
- 超过 70% 的 POI 有名字

---

### Task 2：查询 OSM 数据质量指标

**文件：** `utils/ohsome_quality.py`（Python 脚本，新建）

**在干什么：**
不下载具体 POI，而是查询一些"统计数字"来了解 OSM 数据的质量情况。

**具体怎么做的：**
向 ohsome API 发送 3 个查询：

1. **POI 总数随时间变化**（2020 年 1 月到 2026 年 4 月，按月统计）
   - 如果总数一直在涨 → 说明有人在持续录入新 POI，数据越来越完整
   - 如果趋于平稳 → 说明数据基本饱和

2. **活跃编辑者数量随时间变化**
   - 如果编辑者越来越多 → 数据质量在改善
   - 如果在减少 → 数据可能过时了

3. **创建/修改/删除的比例**
   - 如果"创建"很多 → 说明很多新 POI 被添加
   - 如果"删除"很多 → 说明有清理活动

**输出：** `data/ohsome_quality_results.json`——一个 JSON 文件，包含上述统计结果。

**这些数据有什么用？**
- 写论文的时候，在"数据质量"部分可以引用这些数字
- 如果发现某个区域的数据质量特别差，可以标注出来提醒读者

### Task 3：LDA 主题建模（核心任务，最大的一块）

**文件：** `scripts/part3_poi_lda_yk.Rmd`（R 脚本，新建，**独立开发文件**）

**在干什么：**
这是 Part 3 的核心。把下载的 POI 数据喂给 LDA 算法，让它自动发现新加坡交通站点的城市功能类型。

**"独立开发文件"是什么意思？**
- 我们不会直接在 `report.Rmd`（最终报告）里写代码
- 而是先创建一个独立的 `.Rmd` 文件，在里面开发、测试、调试
- 这个文件能自己 knit（编译）出 HTML 页面，方便查看结果
- 全部验证通过后，再把相关代码复制到 `report.Rmd` 里
- **这样做是为了避免多人编辑同一个文件时产生冲突**

**这个文件包含 13 个代码块（chunks），按顺序执行：**

---

**Chunk 1：加载环境和数据**
- 加载需要的 R 包（`sf` 处理地图、`topicmodels` 运行 LDA、`ggplot2` 画图等）
- 加载 Task 1 下载的 POI 数据和 Part 1 的站点数据
- 定义坐标系统（GPS 坐标 vs 新加坡本地坐标）

---

**Chunk 2：给每个 POI 确定一个"主类别"**
- 一个 POI 可能同时有多个标签（比如一家餐厅既有 `amenity=restaurant`，又可能有 `cuisine=chinese`）
- 我们只需要一个主类别
- 按优先级选：`amenity` > `shop` > `leisure` > ... > `historic`
- 排除住宅相关的 POI（我们的研究关注"目的地"，不是"住的地方"）

**打个比方：** 一个人既是程序员又是吉他手，但我们只能给他贴一个标签。优先级是"职业 > 爱好"，所以贴"程序员"。

---

**Chunk 3：准备地图数据**
- 把 POI 的坐标转换成 R 能处理的地图对象（`sf` 对象）
- 把所有公交站和地铁站合并成一个"全站点"列表
- 去重（有些站点既是公交站又是地铁站，位置一样）
- 给每个站点画一个 **500 米的圆形范围**（"缓冲区"）

**为什么是 500 米？** 这是人们步行约 6-7 分钟的距离，是城市规划中常用的"步行可达范围"。Part 1 也是用 500 米，保持一致。

---

**Chunk 4：把 POI 分配给站点**
- 检查每个 POI 落在哪些站点的 500 米范围内
- 如果一个 POI 在两个站点的范围内，把它分配给最近的那个
- 结果：每个 POI 都归属于某个站点

---

**Chunk 5：过滤"词汇表"**
- 统计每个 POI 类别出现在多少个站点
- **删掉太稀有的**：只在不到 5 个站点出现的类别（比如"独木舟码头"只在 2 个站点附近有，没什么分析价值）
- **删掉太常见的**：出现在超过 90% 站点的类别（比如"公交站"到处都是，不能帮助区分站点类型）
- 这一步之后，预计剩下约 80-150 个类别

**打个比方：** 分析英语文章的主题时，"the"、"is"、"and" 出现在几乎所有文章里，对区分主题没帮助，所以要删掉。反过来，某个词只在一篇文章里出现过一次，也没什么统计意义。LDA 里的这一步就是在做同样的事。

---

**Chunk 6：过滤站点**
- 统计每个站点 500 米范围内有多少个 POI
- **删掉 POI 太少的站点**（少于 3 个），因为数据太稀疏，LDA 算不准
- 预计从约 5,300 个站点减少到 1,500-2,500 个

---

**Chunk 7：构建"文档-词频矩阵"**
- 这一步是 LDA 的输入准备
- 构建一个大表格：行 = 站点，列 = POI 类别，单元格 = 该类别在该站点附近出现了多少个

举个例子：

| 站点 | 餐厅 | 便利店 | 学校 | 银行 | 公园 | ... |
|------|------|--------|------|------|------|-----|
| Orchard | 50 | 12 | 0 | 8 | 2 | ... |
| Kent Ridge | 15 | 5 | 3 | 1 | 1 | ... |
| Boon Lay | 25 | 8 | 1 | 3 | 0 | ... |

这个表格就是 LDA 要分析的数据。

---

**Chunk 8：计算每个站点的数据质量分数**
- 利用 Task 1 下载时带回来的 OSM 元数据，给每个站点打分：
  - `poi_count`：附近总共有多少个 POI（越多越好）
  - `unique_contributors`：有多少个不同的人编辑过这些 POI（越多越好，说明数据可靠）
  - `avg_version`：POI 的平均版本号（越高说明被编辑过多次，数据更准确）
  - `pct_named`：有多少比例的 POI 有名字（越高越好）
  - `pct_version_1`：有多少比例的 POI 从来没被编辑过（越低越好）

**为什么算这个？** 后面如果 LDA 给某个站点分了个奇怪的主题，我们可以看看这个站点的质量分数——如果分数很低，说明可能是数据不够好导致的，不是真的有问题。

---

**Chunk 9：运行 LDA 模型**
- 用 5 个不同的 K 值（4、5、6、7、8）各跑一次 LDA
- K = 我们想让算法找出几个"主题"
- 每次跑 2000 次迭代，让算法充分收敛
- 每个模型保存为一个 `.rds` 文件

**为什么试 5 个 K？** 因为我们不知道新加坡的交通站点到底能分成几类功能。4 太少可能分不够细，8 太多可能分得太碎。试多个值，然后选最好的那个。

---

**Chunk 10：选择最优的 K 值**
- 用一个叫"主题一致性"（topic coherence）的指标来评估每个 K
- 画一张图：横轴是 K 值，纵轴是一致性分数
- 选分数最高的（或者结合人的判断选"最好解释"的那个）

**打个比方：** 你想把一堆衣服分类。分 2 类（上衣/裤子）太粗了。分 20 类（红T恤、蓝T恤、黑裤子...）太碎了。分 5-6 类可能刚刚好（上衣、裤子、外套、鞋子、配饰）。LDA 的 K 选择就是在找这个"刚刚好"的数字。

---

**Chunk 11：解读主题**
- 看每个主题里概率最高的 10-15 个 POI 类别
- 根据这些类别，人工给主题起名字

举个例子，如果主题 3 的 top POI 是：
1. 餐厅 (restaurant) — 12%
2. 咖啡馆 (cafe) — 8%
3. 酒吧 (bar) — 6%
4. 快餐 (fast_food) — 5%
5. 便利店 (convenience) — 4%
...

那我们就给这个主题起名叫 **"餐饮零售区"**。

---

**Chunk 12：给每个站点分类**
- 用选定的 K 值的 LDA 模型，给每个站点算出"属于每个主题的概率"
- 选概率最高的那个主题，作为该站点的"主导功能"
- 同时记录一个"纯度分数"（最高概率有多大，越接近 1 说明这个站点的功能越单一明确）

**举个例子：**
- Orchard：餐饮零售区 65%，商业办公区 20%，休闲娱乐区 15% → 主导：餐饮零售区，纯度 0.65
- 某个偏远公交站：所有主题都差不多各占 20% → 纯度很低，说明这个站点的功能很混合（或者数据太少）

---

**Chunk 13：画图**
- 画 2-3 张（最多 4 张） publication-quality 的图：
  1. **POI 类别分布图**：柱状图，显示最常见的 20 种 POI
  2. **数据质量地图**：新加坡地图，每个站点按质量分数着色
  3. **主题-POI 热力图**：每个主题和它的 top POI 类别的概率
  4. **站点功能地图**（最重要的图）：新加坡地图，每个站点按主导功能着色

所有图都用项目统一的配色方案（`project_colors`）。

```
第 1 步  检查环境和数据是否就绪
   ↓

   ↓
第 3 步  运行 utils/ohsome_quality.py → 得到 ohsome_quality_results.json
   ↓

   ↓
第 5 步  一直调试、运行、看结果，直到所有图和数据都正确

   ↓
第 7 步  把结果整合进 report.Rmd 的 Part 3 部分
   ↓
第 8 步  在群里通知，合并到 main，共享数据文件
```
