## Single Rmd File — Strict Rules

We share ONE `report.Rmd` file. This means merge conflicts are the #1 risk.
Follow these rules to avoid breaking each other's work.

### Section Ownership
- Each member owns ONLY their assigned section (see Part assignments below)
- Never edit another member's section without asking them first
- The shared setup chunk (library loading, data paths) is owned by Part 1
- Part 5 owns the YAML header, abstract, and references section

### Section Markers
Use these comment markers to delimit your section. DO NOT modify markers.
```r
<!-- ===== PART 1: DATA ===== -->
... your code and text ...
<!-- ===== END PART 1 ===== -->
```

### Preventing Merge Conflicts
- ALWAYS `git pull` before starting work
- Commit frequently with small, focused changes
- If you get a conflict, STOP and ask the group chat — do not force push or guess
- Only one person should edit the shared setup chunk at a time
- If you need to add a new package to the setup chunk, message the group first

### Editing Workflow
1. `git pull origin main`
2. Make your changes in YOUR section only
3. Knit locally to verify it compiles
4. `git add report.Rmd` → `git commit -m "Part N: description"`
5. `git push origin main`

## Code Style (R)
- Use tidyverse style (pipes `%>%`, consistent naming)
- Use meaningful variable names (no single letters except `i` in loops)
- Add comments for non-obvious logic
- Don't duplicate code — if someone already wrote a utility function, reuse it
- Keep code chunks focused (one chunk = one logical step)
- Use chunk labels: `{r part1-load-data}`, `{r part2-hourly-plot}`, etc.
- Set `message=FALSE, warning=FALSE` on all chunks (suppress clutter in PDF)
- Hide code in final output where appropriate: `echo=FALSE` for data loading, `echo=TRUE` for analysis

## Terminology (IMPORTANT)
These are established research terminology rules:
- Use "university community" — NOT "students" (data includes staff, visitors, residents)
- Use "travel to" — NOT "seek" or "activate"
- Use "suggest" or "indicate" — NOT "reveal"
- Use "travel" — NOT "play" or "leisure"

## Figure Guidelines
- Use ggplot2 for all figures
- Use consistent color scheme across all parts (agree at kickoff meeting)
- Include axis labels, titles, and legends
- Use clear fonts (min 10pt for labels)
- Set consistent figure size in chunk options: `fig.width=8, fig.height=5`

## Git Rules
- NEVER commit data files (CSV, JSON, parquet, etc.)
- NEVER commit files larger than 10MB
- NEVER force push
- Use descriptive commit messages: "Part 2: add hourly pattern heatmap"
- Work on `main` branch — no feature branches (too complex for one file)
- If stuck on a merge conflict, ask the group chat immediately
